# CODE-RULES.md — Правила точной работы с кодом HayHome

> «Чиним А — проверяем Б. Меняем одно — проверяем связи. Не деплоим — пока не проверим build и тесты.»

---

## 🔴 БЛОК 1: Перед любым изменением кода

### Правило 1.1: Прочитать файл целиком
- Прочитать **весь файл** через `read` перед изменением
- Не редактировать по «примерно помню, что там»
- Проверить: есть ли в этом же файле другие использования изменяемой функции/переменной/компонента

### Правило 1.2: Найти все usage'ы (grep)
- Перед изменением функции/компонента/типа — `grep -r "Name" src/`
- Если меняешь API-роут — найди кто его вызывает (клиентский код, fetch)
- Если меняешь тип — проверь все usage'ы
- Если меняешь ключ переводов — проверь все `tr.key` использования

### Правило 1.3: Проверить БД-схему
- Если меняешь `select()` или данные — сравни с реальной схемой Supabase
- Не включай колонки, которых нет в БД
- Сверяй типы TypeScript с реальными колонками

### Правило 1.4: Не менять без контекста
- Не менять компонент, не проверив родительские/дочерние компоненты
- Не менять API роут, не проверив какие страницы его вызывают
- Не менять middleware, не проверив все matcher-ы

## 🟠 БЛОК 2: После изменения

### Правило 2.1: TypeScript check (обязательно)
```bash
cd /home/n8n/websites/hayhome
npx tsc --noEmit --pretty 2>&1
```
- 0 ошибок — только тогда можно считать изменение готовым
- Если есть ошибки — исправь ВСЕ до того как сказать «готово»

### Правило 2.2: Build check
```bash
npm run build
```
- НЕ деплоить если build не проходит
- Если build прошёл, но с warning-ами — проверить каждый warning

### Правило 2.3: Runtime check (браузер)
- Открыть изменённую страницу в браузере
- Проверить: не 404, не пустая, не битая
- Проверить: кнопка работает (куда ведёт?)
- Проверить: данные загружаются
- Проверить: нет консоль-ошибок

### Правило 2.4: Playwright тест
- Если изменил функциональность — запустить `npx playwright test`
- Если теста нет на эту функциональность — **создать** его
- Каждый баг-фикс = новый тест, который ловит этот баг

## 🔵 БЛОК 3: Структура и организация

### Правило 3.1: Не сломай соседнее
- После изменения — проверить **соседние страницы** (связанные роуты)
- Пример: починил `/hosts` → проверь `/hosts/[id]` и `/book/[id]`
- Пример: починил login → проверь register и forgot-password

### Правило 3.2: Роли и состояния
- Проверить: работает ли изменение для всех ролей (admin, host, guest)
- Проверить: работает ли при пустых данных (0 бронирований, 0 хостов)
- Проверить: работает ли при ошибке (401, 403, 500)
- Проверить: работает ли без localStorage/cookies (чистый браузер)

### Правило 3.3: Языки (10 языков)
- Если меняешь текст — проверь все 10 языков (ru/en/hy/fr/de/es/it/ar/zh/fa)
- Если добавил ключ — проверь что он есть во всех языковых блоках
- Если убрал ключ — проверь что его никто не использует

### Правило 3.4: Коммиты
- Каждое изменение — commit с осмысленным описанием
- Commit должен содержать ЧТО и ПОЧЕМУ
- Перед commit: `git diff --stat` — проверить что меняется
- Не коммитить: лишние файлы, bak-файлы, большие сгенерированные отчёты

## 🔥 БЛОК 4: Известные повторяющиеся паттерны ошибок

| Паттерн | Симптом | Как не повторять |
|---------|---------|-----------------|
| **select(*) → select(cols)** | «0 хостов», пустые страницы | Сверить список колонок с БД |
| **Кнопка не сбрасывает все стейты** | Фильтр остаётся после «Очистить» | Перечислить ВСЕ стейты при сбросе |
| **Починил А → сломал Б** | Сайт работает не на всех страницах | После фикса — проверить соседние роуты |
| **Сказал «работает» без build** | 404, TS-ошибки, рантайм-краши | **Запусти build + tsc + браузер** |
| **Sub-agent сделал не то** | Код изменён не так как надо | Верифицировать результат sub-agent'а |
| **Не проверил типы с БД** | TS ругается, или рантайм ошибка | Сверять типы с реальной БД-схемой |
| **Переводы не на всех языках** | Пустые строки на fr/de/es и т.д. | После добавления ключа — проверить все 10 языков |
